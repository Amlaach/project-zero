## Bug found and fixed during this review (not a design nit — a real defect)

The live tok/s indicator's `\r`+overwrite was clobbering the actual streamed response text, because both shared the same terminal row with no forced newline between them — every stats update erased the response tokens printed so far. This was only caught by looking at a real screenshot, not by the unit tests (which test formatting logic, not terminal-row interaction). Fixed in `src/cli/live_stats.c` by saving the cursor, jumping to the terminal's last row, printing the stats there, and restoring the cursor — a standard "status line" technique that never touches the response's own cursor position. See `docs/ai/mistakes.md`/`change-trace.md` (2026-07-15) for the full writeup.

## Addendum (2026-07-16) — CLI startup banner

A gap identified after the initial review: leading CLI/LLM tools (e.g. Claude Code) show an
animated ASCII-art name banner on startup; project-zero had none. Added `src/cli/banner.c` —
a hand-crafted 5-row block font composing "PROJECT ZERO" letter-by-letter, revealed with a
bottom-up "slide up" animation (~45ms/frame), gated on `is_tty` (never leaks escape codes into
piped output) and shown only for human-watched invocations (REPL, `--server`) — suppressed for
one-shot `--prompt` runs, matching Claude Code's own interactive-vs-scripted banner behavior.

| Principle | Verdict | Notes |
|---|---|---|
| Legible at target size | ✅ | See `06-cli-startup-banner-2026-07-16T01-03-52Z.png` — block letters read clearly as "PROJECT ZERO" once rendered correctly (see bug below). |
| Non-TTY / piped safety | ✅ | `tn_banner_print` no-ops entirely when `is_tty` is false; no escape codes ever leak into redirected output. |
| Suppressed for scripted use | ✅ | Not shown when `--prompt` is passed (one-shot/scripted path), consistent with the "don't slow down scripted invocations" anti-pattern in the principles doc. |
| Color reserved for status | ✅ | Uses the same accent green as the rest of the CLI's bold/color scheme — no new gratuitous color introduced. |

### Bug found and fixed during this review

`compose_banner()` inserted a literal `' '` (space) character between letter glyphs as a
column separator. `print_glyph_row()` only treats the glyph-data `'.'` character as blank,
rendering anything else (including that literal space) as `'#'` — so every inter-letter gap
rendered as a solid filled block instead of a blank column, garbling the banner into an
unreadable wall of `#`. Caught by a raw `script(1)` capture and confirmed by hand-tracing the
composed row string against the expected glyph layout. Fixed by using `'.'` (blank) as the
separator instead of `' '`, consistent with how blank space is represented everywhere else in
the glyph data. See `docs/ai/mistakes.md` (2026-07-16) for the full writeup.
